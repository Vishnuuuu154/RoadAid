# RoadAid Database Design

**Version:** 1.0

**Status:** Draft

---

# Database Overview

The RoadAid database is designed using a relational model to support authentication, roadside assistance, workshop appointments, quotations, payments, reviews, and administration.

The database follows normalization principles to minimize redundancy and maintain data integrity.

# Table: users

## Description

Stores authentication information for all users of the RoadAid system.

This table is shared by:

- Customer
- Service Provider
- Administrator

Role-specific information is stored in separate tables.

| Column | Data Type | Constraints | Description |
|---------|-----------|-------------|-------------|
| id | BIGINT | Primary Key, Auto Increment | Unique user ID |
| full_name | VARCHAR(100) | NOT NULL | Full name |
| email | VARCHAR(100) | UNIQUE, NOT NULL | Email address |
| phone | VARCHAR(15) | UNIQUE, NOT NULL | Mobile number |
| password | VARCHAR(255) | NOT NULL | Encrypted password |
| role | VARCHAR(30) | NOT NULL | CUSTOMER, SERVICE_PROVIDER, ADMIN |
| profile_image | VARCHAR(255) | NULL | Profile image path |
| is_active | BOOLEAN | DEFAULT TRUE | Account status |
| created_at | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | Account creation time |
| updated_at | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | Last update time |

**Primary Key**

- id

**Unique Keys**

- email
- phone

# Design Decisions

- One users table for authentication.
-Customer information is stored directly in the users table. 
- Service-provider-specific information is stored separately because providers have additional attributes such as experience, verification status, ratings, completed jobs, and online status.
- Workshop information is stored separately.
- One quotation per request.
- Fuel Assistance only provides nearby fuel station information.
- Passwords will be stored using BCrypt hashing.
- Images are stored as file paths, not binary data.
- Status values use VARCHAR instead of ENUM.

# Design Rationale

The database follows normalization principles while avoiding unnecessary tables.

- Authentication data is stored in the `users` table.
- Customer-specific data is stored in feature-specific tables such as `vehicles`, `favorites`, `reviews`, and `complaints`.
- Service-provider-specific data is stored separately because providers have additional business attributes.
- Workshop information is separated because not every service provider owns a workshop.
- One quotation is generated for each service request.
- Status values are stored using `VARCHAR` for flexibility.
